HOP · the orchestrator

One binary runs the cluster.

Agent, leader and HTTP API in one Go binary — no database, no sidecars. Jobs with health checks, restarts and priorities; an HMAC-signed API; leader election on a CAS lock with cluster state in any S3 bucket — a node that reboots brings its own jobs back. And it runs every kind of workload: processes, containers, whole machines.

Every workload. Three drivers, one job spec.

exec

Plain processes with isolation and limits, on the Linux and macOS machines you already have. Adopt HOP today without touching your OS choice.

Docker

Your existing containers, unchanged — same job spec, same API. HOP schedules and supervises; Docker stays the runtime you know.

hop → HopOS

The workload the others can’t: apps without an OS underneath. Whole cores, hardware cages, per-app cpu/mem/temp telemetry with no agents or exporters — see the HopOS story.

State lives in any S3 bucket — one committed object is the whole database, and the leader itself fails over on the same lock (~60 s). Reboot a node, it reloads its own jobs: 126/126 back in 17.4 s after a hard power cut, zero resubmits, measured on real hardware. count: -1 runs a job on every node in the cluster.

What HOP is not

Not Kubernetes: no CRDs, no operators, no YAML sprawl — and none of the ecosystem either. What it is: a full cluster — leader election, agent heartbeats, cross-node dispatch, ~60 s failover, federation — that has been running our own production workloads for months. We eat here first.

Better together. Start with HOP on the machines you have; add HopOS nodes where isolation and simplicity pay the most — the edge box, the appliance, the machine that must survive a power cut.
