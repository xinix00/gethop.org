HopOS · the Go-only OS

No kernel between your app and the hardware.

A ~15 MB signed image on a USB stick is the entire node. It boots any UEFI arm64 box or a Raspberry Pi, and runs each Go app on its own physical cores, inside its own memory cage. Your core count is your headroom — same image, Pi to 128-core server.

Security by subtraction

Meltdown, Spectre, MDS, container escapes — different bugs, one root: sharing. Linux must safely share everything between everyone, with mitigations that are partial, ongoing and expensive. HopOS takes the simpler road: share nothing by default. Cores and memory are handed out whole, and the cage is the silicon’s second translation stage — not kernel policy.

Attack classLinux answer (mitigation)HopOS answer (structure)
Meltdown-class: kernel leaks to userKPTI — unmap the kernel, mostlyNo kernel in the app’s address space at all. Behind stage-2, the OS is unnameable.
SMT side channels (MDS family)Disable hyperthreading, scrub buffersBy default, apps never share a core — not even in time.
Spectre across processesFlush branch predictors on every context switchBy default there is no context switch — your neighbour never runs on your core.
Container escapes (runc, Dirty Pipe, …)Patch treadmillNo containers, and no shared kernel to escape into.
~400 syscalls + ioctl jungleseccomp/AppArmor profiles you must writeAn ABI of a handful of operations, enforced by hardware, not policy.
Rogue DMA from a workloadIOMMU configuration, fingers crossedApps have zero MMIO. Only the node programs devices.

The double win: no mitigations tax. Apps run at the silicon’s spec clock — measured, not promised — because there is nothing to scrub. You don’t have to patch what you don’t share.

Safe by default, share when trusted. Everything above is the default: every app owns whole cores. When density matters more than a private core, you opt in — tag apps with a sharegroup and they cooperatively share a pool of whole cores you size yourself. You’re naming your own cage-mates: inside the group the timing isolation (the MDS / Spectre rows) is traded for density, like any shared core — but the memory and network cage never softens. Each app keeps its own stage-2 partition and its own network stack, so nothing bleeds between neighbours, and apps you don’t group keep the whole-core guarantee in full. It’s how a 4-core Pi runs a dozen trusted services, or a microservice fleet packs onto a handful of cores — without giving up hardware isolation between tenants.

Nothing lives on the node

HopOS is stateless — and so is HOP on top of it. The boot stick carries a signed image and six lines of config; jobs, app images and cluster state live in object storage and are fetched at boot. State on S3, not on metal — there is nothing on the machine to lose, corrupt, drift or steal.

That is the whole trick behind the power-cut number: pull the plug on 126 running jobs and the node rebuilds itself from S3 in 17.4 s. Lose the machine entirely? Move the stick to another box. “Reinstall” means re-imaging a USB stick. Even shared data is designed to be fetched, not owned.

Small enough to actually read

Lines of code, excluding tests, comments and blanks — counted by scripts in the repo, rounded so the numbers stay true across builds. The whole OS fits in a single AI context window: audit it in one sitting, human or machine. The rung you have to trust is the first one; drivers and board support are swappable outer layers, already outside every cage.

A Linux node doing the same job trusts GRUB, the kernel (~30,000,000 lines), systemd, libc and a container runtime. HopOS is the whole node — bootloader included — in ~11,600.

HopOS does less. On purpose.

It runs Go applications, compiled for HopOS — no containers, no other languages, no general-purpose anything. That narrowness is where the isolation and the ~15 MB come from. Need Docker images or non-Go services? Run HOP with its exec and Docker drivers on a Linux box in the same cluster, and keep HopOS for the apps that deserve the metal.

Boot a node in five minutes

No hardware needed — the same image that runs a 128-core Ampere Altra boots in QEMU. Full guides: the docs; signed images: releases.

 in QEMU — no hardware
# needs: qemu-system-aarch64 + tamago-go
$ git clone https://github.com/xinix00/HopOS
$ cd HopOS && ./image/uefi-run.sh agent
hop: agent starting — HOPOS_AGENT_UP
 on real metal — stick or SD card
# signed images: github.com/xinix00/HopOS/releases
# UEFI arm64 box — USB stick
$ cp BOOTAA64.EFI /Volumes/STICK/EFI/BOOT/
$ $EDITOR /Volumes/STICK/hopos.cfg      # 6 lines
# Raspberry Pi 4/5 — SD card (firmware boot, no UEFI)
$ cp hop-agent5.img config.txt /Volumes/bootfs/
$ $EDITOR /Volumes/bootfs/cmdline.txt   # same hopos.* keys
# plug in, power on. that’s the install.

Better together. HopOS nodes join a HOP cluster next to your Linux boxes — a caged app on metal and a Docker job on Linux are the same job spec, the same API.
